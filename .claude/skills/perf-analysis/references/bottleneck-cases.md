# 典型瓶颈诊断案例

## 案例 1：标量执行主导（逐元素算子）

| 指标 | 典型值 | 判定 |
|------|--------|------|
| vec_ratio | < 10% | ❌ 严重 |
| scalar_ratio | > 60% | ❌ 严重 |
| icache_miss | > 15% | ❌ |
| vec_wait | < 5% | ✅ |
| mte2_wait | 0% | ⚠️ 串行信号 |

**诊断**：vec_ratio 极低 + scalar_ratio 极高 → 标量执行为主，向量单元严重闲置。mte2_wait=0% 结合高 scalar_ratio → 未做流水线重叠，搬运与计算串行。

**策略**：
1. (高) 确认 Compute() 是否全部使用 vec_* 批量 API
2. (高) 流水线重叠（预取→计算→写回三级流水）
3. (中) 循环展开减少 icache_miss
4. (高) 若 BlockDim=1 → 启用多核

## 案例 2：搬运瓶颈

| 指标 | 典型值 | 判定 |
|------|--------|------|
| vec_ratio | ~45% | ⚠️ |
| mte2_ratio | > 30% | ❌ |
| mte3_ratio | > 15% | ⚠️ |
| mte2_wait | > 8% | ❌ |
| mte2_active_bw | 远低于峰值 | ❌ 搬运效率低 |

**诊断**：搬运占比高 + 向量等待搬运 → 未做流水线重叠。

**步骤**：
1. 先计算理论搬运耗时 = 搬运量(GM_to_UB_datas) / GM 峰值带宽
2. 若实际 ≈ 理论 → MTE2 已达带宽上限，优化方向是流水线重叠
3. 若实际 >> 理论 → 搬运效率低，检查对齐和搬运粒度

**策略**：
1. (高) Process() 改为预取-计算-写回三级流水
2. (中) 增大 UB_FORMER 减少搬运次数
3. (低) 512B 地址对齐、增大单次搬运量 ≥ 16KB

## 案例 3：Bank 冲突

| 指标 | 典型值 | 判定 |
|------|--------|------|
| bankgroup_cflt | > 2% | ❌ |
| vec_wait | ~5% | ⚠️ |

**诊断**：多个 buffer 的 UB 地址映射到同一 bank group。通常发生在多维数据访问模式中。

**策略**：
1. (高) UB buffer 分配时添加 padding 偏移打破 bank 对齐
2. (中) 改变访问步长，避免多线程同时访问同一 bank
3. (中) 调整数据布局（交错存储）

## 案例 4：多核启用但性能下降

| 指标 | BlockDim=1 | BlockDim=N | 变化 |
|------|-----------|-----------|------|
| Task Duration | 基准 | 略降 | 少量改善 |
| scalar_ratio | 基准 | **大幅上升** | 每核数据量变小，头开销占比增大 |
| vec_wait | 基准 | **大幅上升** | 多核竞争带宽 |
| L2 hit | 基准 | **大幅下降** | 每核数据量不足 L2 复用 |
| mte2_wait | 0% | **> 10%** | 多核并发搬运竞争 GM 带宽 |

**诊断**：多核加速比远低于核心数。每核数据量过小导致头开销和并发竞争抵消了并行收益。**多核 ≠ 加速多**，需要足够的总数据量支撑多核扩展。

**策略**：
1. 每核至少 2048 元素（Elementwise），不足则减少核数
2. 数据量 < 16KB 时不建议启用多核
3. 关注核间负载均衡（尾核特殊处理）
